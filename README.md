## 1. 직무 관련 핵심 역량 요약

NHN AX개발팀은 임직원이 사용하는 협업 시스템을 개선하고 기업의 AI 전환을 돕는 역할을 수행한다고 이해했습니다. 협업 시스템의 도메인을 파악한 뒤 사용자에게 필요한 기능을 UI/UX와 백엔드 서비스로 연결하고, 시스템 연동 과정에서 발생하는 문제를 개선 근거로 정리하는 역량이 중요하다고 판단했습니다.

AI 기능을 단순한 모델 출력으로 끝내지 않고 사용자의 요청, 데이터 검색, Tool 실행, 결과 저장, 화면 조회까지 이어지는 서비스 흐름으로 연결하는 프로젝트를 수행해 왔습니다. 특히 LLM, RAG, Agentic AI를 활용해 반복 업무를 자동화하고 결과를 확인 가능한 산출물로 제공하는 구조를 구현했습니다.

| 역량 | 관련 경험 |
| --- | --- |
| LLM/RAG 기반 업무 자동화 | 문서 검색, 코드 분석 결과, LLM 응답 생성을 연결해 개발자가 활용 가능한 리포트 생성 |
| Agentic AI 기반 Tool 실행 | 자연어 요청을 분석 Tool 실행 흐름으로 변환하고 필요한 Tool과 인자를 선택하는 구조 구현 |
| Backend API 연동 | FastAPI 기반 CSV 업로드, 워크플로우 실행, 결과 조회 API 구성 |
| 백엔드 오케스트레이션 | Spring Boot, Kafka, 작업 ID 관리, 결과 조회 흐름을 활용한 요청·응답 흐름 관리 |
| 협업 및 문서화 | 오류 해결 과정, 실행 방법, 산출물 경로를 정리해 팀원이 동일한 환경에서 테스트 가능하도록 문서화 |

## 2. Project 01. LLM/RAG 기반 eGovFrame 코드 변환·보안 자동화 에이전트

### 2.1 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 기간 | 2025.06 ~ 2025.09 |
| 팀 규모 | 8명 |
| 주요 기술 | Python, LangChain, LangGraph, FAISS, RAG, SonarQube API, Spring Boot, Kafka, Docker, Kubernetes, Azure, eGovFrame |
| 수상 | KT AIVLE School BigProject 최우수상 |
| 담당 역할 | LangGraph 기반 RAG 보안 검토 파이프라인 구현, Spring Boot 기반 백엔드 오케스트레이션 설계 |
| 성과 | Dual Retrieval과 Reranking 구조로 코드 맥락 누락을 개선하고 약 95% 이상의 유효 리포트 비율 확보 |

Python 기반 레거시 코드를 전자정부 표준프레임워크 구조로 변환하고 SonarQube 분석 결과를 기반으로 보안 취약점 개선 방향을 제안하는 자동화 에이전트 시스템입니다. 코드 분석, 보안 진단, 가이드 문서 검색, 권장 코드 생성, 리포트 저장까지 이어지는 End-to-End 자동화 흐름을 목표로 했습니다.

### 2.2 문제 정의

전자정부프레임워크 전환 과정에서는 기존 소스 구조를 분석하고 Controller, Service, DAO 등 프레임워크 규칙에 맞게 구조를 재배치해야 합니다. 또한 공공 시스템 적용을 위해 보안 품질을 함께 고려해야 하므로 변환된 코드에 대한 취약점 진단과 개선 가이드 제공이 필요했습니다.

해결해야 할 문제는 다음과 같았습니다.

- 코드 구조와 프레임워크 규칙을 사람이 직접 분석해야 해 시간과 비용이 증가함
- SonarQube 결과만으로는 개발자가 참고해야 할 보안 가이드 조항을 바로 파악하기 어려움
- 일반적인 벡터 검색만 적용하면 특정 키워드에 결과가 편향되고 코드 맥락이 충분히 반영되지 않음
- 분석 결과가 서비스 흐름과 분리되면 개발자가 반복적으로 활용하기 어려움

### 2.3 시스템 흐름

```text
Source Code Upload
  → Project Structure Analysis
  → Language / Framework Conversion Agent
  → eGovFrame Structure Mapping
  → SonarQube Security Analysis
  → Dual Retrieval 기반 보안 가이드 검색
  → Reranking 및 누락 방지 보강
  → LLM 기반 권장 코드 및 개선 리포트 생성
  → Backend API를 통한 결과 조회
```

보안 에이전트는 SonarQube 이슈 메시지와 관련 코드 스니펫을 입력으로 받아 보안 가이드 문서와 코드 예시를 검색합니다. 이후 개발자가 참고할 수 있는 개선 리포트를 생성하고 저장하는 역할을 담당했습니다.

### 2.4 주요 구현 내용

#### 2.4.1 Dual Retrieval 기반 RAG 구조 설계

보안 가이드 문서를 그대로 벡터 검색에 활용했을 때 특정 키워드 중심으로 검색 결과가 편향되는 문제가 있었습니다. 이를 개선하기 위해 문서의 텍스트 조항과 코드 예시를 분리하고 각각 FAISS 기반 인덱스로 구성했습니다.

| 검색 채널 | 목적 |
| --- | --- |
| 텍스트 조항 인덱스 | 보안 가이드의 의미, 적용 조건, 설명 중심 검색 |
| 코드 예시 인덱스 | SonarQube 이슈 메시지와 코드 스니펫을 반영한 코드 맥락 중심 검색 |

이 구조를 통해 이슈의 의미와 실제 코드 맥락을 함께 고려할 수 있도록 했습니다.

#### 2.4.2 Reranking 및 누락 방지 로직 추가

두 검색 채널에서 나온 결과를 그대로 사용하면 일부 중요한 가이드 섹션이 누락될 수 있었습니다. 검색 결과를 재정렬하고 핵심 가이드 섹션이 빠지지 않도록 보강 로직을 추가했습니다.

검토 기준은 다음과 같이 설정했습니다.

- SonarQube 이슈와 보안 가이드 조항의 관련성
- 코드 스니펫과 검색된 코드 예시의 맥락 일치 여부
- 최종 리포트에 핵심 보안 가이드가 포함되었는지 여부
- 개발자가 참고 가능한 수준의 권장 코드가 생성되었는지 여부

#### 2.4.3 LangGraph 기반 상태 워크플로우 구현

보안 진단 과정은 한 번의 LLM 호출이 아니라 여러 단계의 상태를 거쳐야 했습니다. LangGraph 기반으로 다음 노드를 구성했습니다.

```text
build_query
  → retrieve_text
  → retrieve_code
  → fuse_scores
  → section_coverage
  → generate_guides
  → save_outputs
```

각 단계는 이슈 분석, 검색, 결과 통합, 누락 점검, 리포트 생성, 산출물 저장의 역할을 맡도록 분리했습니다. 이를 통해 분석 흐름을 추적하기 쉬웠고 문제가 발생했을 때 오류 지점을 단계별로 확인할 수 있었습니다.

#### 2.4.4 Spring Boot 기반 백엔드 오케스트레이션

RAG 결과를 서비스 기능으로 제공하기 위해 Spring Boot 기반 MSA 백엔드와 연결했습니다. 분석 요청 생성, 작업 ID 관리, 결과 조회 흐름을 구현하고 Kafka 헤더 기반 라우팅을 적용해 요청과 결과 흐름을 관리했습니다.

```text
Frontend 요청
  → Backend 분석 요청 생성
  → Kafka Topic으로 Agent 요청 전달
  → Python 기반 Agent 실행
  → 결과 저장
  → Backend 결과 조회 API 제공
  → Frontend에서 리포트 확인
```

### 2.5 검증 방식 및 성과

SonarQube 이슈별 생성 리포트를 대상으로 다음 항목을 검수했습니다.

- 보안 가이드 조항과의 매칭 여부
- 핵심 가이드 누락 여부
- 권장 코드의 적용 가능성
- 리포트 내용의 일관성

검수 결과 약 95% 이상의 유효 리포트 비율을 확보했습니다. 문서 검색, 코드 분석, 백엔드 요청 흐름, 결과 리포트 생성을 하나의 서비스 기능으로 연결하며 AX 개발 직무와 맞닿은 경험을 쌓았습니다.

### 2.6 NHN AX개발팀 업무와의 연결점

이 프로젝트는 개발자가 반복적으로 수행해야 하는 코드 분석과 보안 검토 작업을 자동화하고 결과를 리포트로 제공했다는 점에서 AX 업무와 연결됩니다. 협업 시스템에서도 반복 업무를 분석하고 필요한 기능 실행과 결과 확인 흐름을 자동화하는 구조가 중요하다고 생각합니다.

활용 가능한 경험은 다음과 같습니다.

- 협업 시스템의 반복 업무를 분석하고 자동화 흐름으로 전환한 경험
- 외부 분석 도구와 LLM/RAG 구조를 연결한 경험
- 백엔드 요청, 작업 ID, 결과 조회, 리포트 저장까지 고려한 서비스 흐름 설계 경험
- 문제 발생 지점을 단계별로 분리하고 개선 근거를 쌓은 경험

## 3. Project 02. LLM Agent 기반 Tool Calling 분석 워크플로우 자동화 시스템

### 3.1 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 기간 | 2025.07 ~ 2025.11 |
| 팀 규모 | 2명 |
| 주요 기술 | Agentica, OpenAI API, TypeScript, Python, FastAPI, Node.js, Pandas, scikit-learn, XGBoost |
| 수상 | 오픈소스 개발자대회 우수작 |
| 담당 역할 | Tool Calling 기반 분석 파이프라인 설계, Direct Execute 실행 모듈 구현 및 FastAPI 연동 |
| 성과 | 58개 자연어 분석 요청 시나리오 중 50개 요청에서 적절한 Tool을 선택해 86.21%의 라우팅 정확도 확인 |

데이터 분석 전문 지식이 부족한 사용자도 자연어 명령만으로 정형 데이터 분석을 수행할 수 있도록 돕는 LLM Agent 기반 데이터 분석 자동화 시스템입니다. CSV 업로드부터 데이터 미리보기, 분석 워크플로우 실행, 결과 파일 저장, 화면 조회까지 이어지는 End-to-End 분석 파이프라인을 구성했습니다.

### 3.2 문제 정의

기초적인 데이터 분석은 반복적으로 이루어지지만 실제 실행은 여전히 사람의 수작업에 의존하는 경우가 많습니다. 사용자의 요청마다 필요한 분석 기능과 실행 조건이 다르기 때문에 단순 챗봇처럼 질문에 답하는 구조만으로는 분석 Tool 실행까지 연결하기 어렵습니다.

해결해야 할 문제는 다음과 같았습니다.

- 자연어 요청을 실제 분석 Tool 실행으로 연결해야 함
- 요청마다 필요한 Tool과 실행 인자가 달라 동적인 판단 구조가 필요함
- 전체 분석 과정을 매번 LLM 판단에만 맡기면 실행 순서와 결과가 흔들릴 수 있음
- 분석 결과 파일, 시각화 이미지, 모델 결과, 리포트를 화면에서 확인할 수 있어야 함

### 3.3 시스템 흐름

분석 시스템은 유연성이 필요한 대화형 분석과 재현성이 필요한 전체 워크플로우를 분리해 설계했습니다.

```text
[대화형 분석]
사용자 자연어 요청
  → Agent가 의도와 데이터 상태 분석
  → 필요한 Tool 및 인자 선택
  → Tool 실행
  → 결과 저장
  → 응답 및 화면 조회

[워크플로우 기반 분석]
CSV Upload
  → Data Preview
  → Basic Statistics
  → Preprocessing
  → Visualization
  → Modeling
  → Report Generation
  → Result View
```

사용자 요청에는 동적으로 대응하면서도 리포트 생성 과정에서는 실행 순서와 산출물 구조의 일관성을 확보했습니다.

### 3.4 주요 구현 내용

#### 3.4.1 Agentica 기반 Function Calling 구조 구성

사용자가 자연어로 분석 요청을 입력하면 LLM이 현재 데이터 상태와 사용자의 의도를 바탕으로 필요한 Tool과 호출 인자를 결정하도록 구성했습니다.

예시는 다음과 같습니다.

```text
사용자 요청: "결측치가 많은 컬럼을 확인하고 시각화해줘"
  → Agent가 요청 의도 분석
  → 데이터 상태 확인 Tool 선택
  → 결측치 분석 Tool 실행
  → 시각화 Tool 실행
  → 결과 요약 및 파일 저장
```

#### 3.4.2 Direct Execute 워크플로우 설계

반복 실행이 필요한 전체 분석 과정은 Direct Execute 방식으로 분리했습니다. 업로드부터 리포트 생성까지 정해진 흐름으로 실행되도록 설계해 LLM 응답 변동성에 따른 실행 불안정성을 줄였습니다.

#### 3.4.3 데이터 상태 기반 Tool 실행 흐름 구성

업로드된 CSV의 구조와 품질 정보를 먼저 추출하고 후속 Tool의 입력으로 전달했습니다. 이를 통해 분석 과정이 사용자의 요청뿐 아니라 실제 데이터 상태를 기반으로 이어지도록 구성했습니다.

확인한 데이터 정보는 다음과 같습니다.

- 컬럼명 및 데이터 타입
- 결측치 여부
- 기초 통계량
- 범주형/수치형 컬럼 구분
- 모델 학습 가능 여부

#### 3.4.4 FastAPI 기반 서비스 흐름 통합

FastAPI를 활용해 분석 시스템의 주요 API를 구성했습니다.

```text
POST /upload_csv      : CSV 업로드
GET  /preview         : 데이터 미리보기
POST /run_workflow    : 전체 분석 워크플로우 실행
POST /chat            : 자연어 분석 요청 처리
GET  /outputs         : 생성 결과 조회
```

TypeScript 기반 Agent 실행 결과와 Python 분석 모듈의 산출물을 연결했습니다. 자연어 요청에서 Tool 실행, 결과 저장, 화면 조회까지 이어지는 분석 파이프라인을 구현했습니다.

#### 3.4.5 결과 파일 저장 및 화면 조회 구조 정리

분석 결과가 일회성 응답으로 사라지지 않도록 표준 저장 경로를 정했습니다. 전처리 파일, 시각화 이미지, 모델 학습 결과, 리포트를 분리해 저장하고 화면에서 단계별로 확인할 수 있도록 연결했습니다.

```text
src/uploads         : 사용자가 업로드한 CSV 파일
src/outputs         : 분석 결과 파일
src/outputs/img     : 시각화 이미지
src/outputs/report  : 결과 리포트
```

### 3.5 검증 방식 및 성과

사전에 정의한 58개 자연어 분석 요청 시나리오를 기준으로 Tool 선택 결과를 검증했습니다. 그중 50개 요청에서 적절한 Tool을 선택해 86.21%의 라우팅 정확도를 확인했습니다.

검증 기준은 다음과 같습니다.

- 사용자의 요청 의도를 올바르게 해석했는지
- 필요한 분석 Tool을 선택했는지
- Tool 실행에 필요한 인자를 적절히 구성했는지
- 실행 결과가 사용자가 기대한 분석 결과와 연결되는지

### 3.6 NHN AX개발팀 업무와의 연결점

이 프로젝트는 자연어 요청을 실제 Tool 실행과 API 응답 흐름으로 연결했다는 점에서 AX 기반 협업 시스템 개선 업무와 연결됩니다. 협업 시스템에서도 사용자의 반복 요청을 파악하고 필요한 기능 실행과 결과 제공을 자동화하는 구조가 중요하다고 생각합니다.

활용 가능한 경험은 다음과 같습니다.

- 사용자 요청을 기능 실행 흐름으로 변환한 경험
- LLM 응답 변동성을 줄이기 위해 동적 실행과 고정 워크플로우를 분리한 경험
- FastAPI 기반으로 업로드, 실행, 결과 조회 API를 구성한 경험
- 결과 파일과 산출물 경로를 표준화해 재현성과 확인 가능성을 높인 경험

## 4. Additional Project. 음성-수어 번역 및 아바타 생성 End-to-End 파이프라인

### 4.1 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 기간 | 2024.03 ~ 2024.10 |
| 팀 규모 | 5명 |
| 주요 기술 | Python, PyTorch, Hugging Face Transformers, KoBART, OpenAI GPT, ComfyUI, Stable Diffusion, ControlNet, DWPose |
| 수상 | 한이음 ICT 공모전 프로보노 부문 입선 |
| 담당 역할 | KoBART 기반 수어 문법 변환 모델 학습 및 아바타 생성 연계 파이프라인 구성 |
| 성과 | GPT 기반 데이터 증강 후 검증 데이터 기준 토큰 단위 F1 Score 0.86 확인, BLEU 4.2p 향상 |

한국어 구어 문장을 수어 문법 표현으로 변환하고 이를 수어 아바타 영상 생성 흐름으로 연결하는 End-to-End 파이프라인입니다. 텍스트 모델 학습에 그치지 않고 변환 결과가 실제 영상 산출물로 이어지는 전체 흐름을 구성했습니다.

### 4.2 문제 정의

일반 한국어 문장을 단어 단위로 변환하는 방식은 수어의 주제 우선 구조, 주체 생략, SOV 어순을 충분히 반영하기 어렵습니다. 수어는 문장의 의미뿐 아니라 동작 순서와 표현 방식까지 함께 전달되어야 하므로 수어 문법에 맞는 텍스트 변환과 영상 생성 단계의 연결 구조가 필요했습니다.

### 4.3 시스템 흐름

```text
한국어 음성·문장 입력
  → STT 및 텍스트 입력 처리
  → KoBART 기반 수어 문법 변환
  → 수어 동작 영상 매칭
  → DWPose 기반 pose keypoint 추출
  → ControlNet 기반 아바타 프레임 생성
  → 영상 산출물 변환
```

### 4.4 주요 구현 내용

#### 4.4.1 KoBART 기반 수어 문법 변환 모델 학습

국립국어원 한국어와 한국수어 병렬 말뭉치를 기반으로 KoBART Seq2Seq 모델을 fine-tuning했습니다. 한국어 문장을 입력하면 대응되는 한국수어 문장이 생성되도록 학습 쌍을 정리했고 모델이 단순 원문 단어 나열이 아니라 수어 표현에 가까운 문장 구조를 생성하도록 데이터를 구성했습니다.

#### 4.4.2 GPT 기반 데이터 증강 적용

입력 표현이 제한되면 실제 사용자의 다양한 구어체 발화에 대응하기 어렵다고 판단했습니다. 동일한 수어 문장에 대해 의미를 유지한 한국어 변형 문장을 생성하여 원본 7,108개 데이터 쌍을 35,540개 규모로 확장했습니다.

#### 4.4.3 아바타 생성 연계 파이프라인 구성

수어 동작 영상에서 pose keypoint를 추출하고 이를 생성 조건으로 활용했습니다. DWPose로 관절과 손 위치 정보를 추출한 뒤 ControlNet을 통해 포즈 조건을 유지한 Stable Diffusion 기반 아바타 생성 워크플로우를 구성했습니다.

### 4.5 검증 방식 및 성과

검증 데이터 기준 토큰 단위 F1 Score 0.86을 확인했고 GPT 기반 데이터 증강 후 BLEU가 4.2p 향상되었습니다. 생성 단계에서는 checkpoint와 LoRA 적용 결과를 비교해 손동작과 인물 디테일이 유지되는지 확인했습니다. 이후 생성된 프레임을 영상 산출물로 변환해 수어 변환 결과가 실제 아바타 영상으로 이어지는지 검증했습니다.

### 4.6 NHN AX개발팀 업무와의 연결점

직무와 직접적인 도메인은 다르지만 AI 모델의 추론 결과를 후속 시스템과 연결해 사용자가 확인 가능한 결과물로 제공했다는 점에서 의미가 있습니다. 입력 처리, 모델 추론, 생성 조건 구성, 결과 산출까지 이어지는 파이프라인을 구성하며 AI 기능을 서비스 흐름으로 연결하는 경험을 쌓았습니다.

## 5. 협업 및 문서화 방식

### 5.1 요구사항 확인 방식

요구사항을 바로 판단하기보다 먼저 정리하고 확인하는 방식으로 협업합니다. 같은 요청도 사용자나 팀원마다 중요하게 생각하는 기준이 다를 수 있기 때문에 기능 요구사항과 제약 조건을 먼저 확인한 뒤 개발 방향을 조율해 왔습니다.

### 5.2 기술 선택 기준

불확실한 기술 선택은 감에 의존하지 않고 검증 가능한 기준으로 판단합니다. LLM 기반 서비스 구조를 설계할 때 외부 API 활용과 오픈소스 모델 파인튜닝 의견이 나뉘었고 응답 품질, 개발 기간, 데이터 준비 부담, 운영 비용을 기준으로 비교했습니다. 이를 바탕으로 제한된 기간 내 핵심 기능 구현을 위해 RAG 구조를 우선 적용하고 모델 튜닝은 데이터 확보 후 고도화 단계로 나누는 방향으로 정리했습니다.

### 5.3 MVP와 문서화

제한된 일정에서는 반드시 동작해야 하는 MVP를 먼저 정의하고 구현 범위를 현실적으로 조정합니다. 개발 과정에서 발생한 오류와 해결 방법은 이슈와 커밋 메시지로 기록했습니다. 필요한 패키지 설치 방법과 실행 순서는 문서화해 팀원들이 동일한 환경에서 테스트할 수 있도록 했습니다.
